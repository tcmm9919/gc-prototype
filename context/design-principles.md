# GCP — Design Principles (для дизайн-ревью)

> Каноничный UX-фреймворк проекта — **[`design.md`](../design.md)** (Designer's Brain: Nielsen, Gestalt, UX-laws, shadcn-подход).
> Этот файл — операционный чек-лист ревьюера: что именно проверять на экранах GCP. При конфликте выигрывает `design.md`.
> Токены и палитра — **[`style-guide.md`](./style-guide.md)** (выжимка из `globals.css`).

GCP — внутренняя AML/KYC-платформа Freedom Bank для комплаенс-офицеров. Это **плотный B2B-инструмент** для ежедневной работы с риском, не лендинг. Приоритет: скорость сканирования, доверие к данным, отсутствие двусмысленности в решениях о риске.

## I. Жёсткие правила проекта (нарушение = Blocker / High)
- [ ] **4 состояния каждого экрана**: Empty / Loading / Data / Error (design.md §D). Прототип сейчас не имеет `loading.tsx` / `error.tsx` / `not-found.tsx` — отмечать отсутствующие состояния как находку.
- [ ] **Один primary CTA на экран**. Множественные зелёные кнопки в одном вью — High-Priority.
- [ ] **Microcopy = глагол + объект** («Добавить продукт», не «Submit» / не «OK»). RU-интерфейс — без англицизмов и кальки.
- [ ] **Только семантические токены** — никаких `bg-white`, hex, inline-стилей (см. style-guide.md).
- [ ] **Иконки** — только `lucide-react`, дефолтный размер `size-4`.

## II. Иерархия и сканируемость (плотные данные)
- [ ] Чёткая визуальная иерархия: размер/вес/цвет ведут глаз к главному (риск-балл, статус, сумма).
- [ ] Таблицы: текст слева, числа справа; жирные заголовки; адекватная высота строки; sticky-заголовки на длинных списках.
- [ ] Консистентное выравнивание и сетка между экранами одного модуля.
- [ ] Щедрый white space на обзорных экранах (дашборд), плотность — в таблицах (Linear precision).

## III. Риск, статусы, доверие к данным
- [ ] Risk-уровни (low/medium/high/critical) консистентны по цвету и подписи во всех модулях; цвет не единственный носитель смысла (дублировать текстом/иконкой — WCAG 1.4.1).
- [ ] Статус-баджи читаемы и единообразны (Pending / Approved / Rejected / Escalated и т.п.).
- [ ] Числа, ID, суммы, даты — единый формат; денежные значения и даты локализованы под RU/KZ-контекст.
- [ ] Деструктивные / эскалационные действия («Поднять риск», «Закрыть кейс») требуют подтверждения.

## IV. Интеракции и моушн
- [ ] Микро-интеракции быстрые (150–300ms, design.md §H), `ease-in-out`, осмысленные — не отвлекают.
- [ ] Hover-lift карточек: `--shadow-card` → `--shadow-card-hover`.
- [ ] Loading: скелетоны для страниц, спиннеры для in-component действий.
- [ ] `prefers-reduced-motion` уважается.

## V. Доступность (WCAG 2.1 AA)
- [ ] Контраст текста ≥ 4.5:1 (особенно `muted-foreground` / `subtle-foreground` на surface и баджи risk-foreground на risk-bg).
- [ ] Полная клавиатурная навигация, видимый focus-state на всех интерактивных элементах.
- [ ] Семантический HTML, label'ы у полей, alt у значимых изображений.
- [ ] Touch-таргеты ≥ 44px на узких вьюпортах.

## VI. Адаптивность
- [ ] Desktop 1440 / tablet 768 / mobile 375 — без горизонтального скролла и наложений.
- [ ] Плотные таблицы корректно деградируют на узких экранах (скролл-контейнер / карточный режим), 88px-сайдбар адаптируется.

## VII. Темы
- [ ] Проверка в **light и dark** (Revolut-style dark). Токены, а не хардкод; primary читается в обеих темах.

---
### Триаж находок
- **[Blocker]** — критично, чинить немедленно (сломанный флоу, нечитаемый риск, провал контраста на ключевом тексте).
- **[High-Priority]** — до мёржа (нарушение жёстких правил §I, нет состояния Empty/Error).
- **[Medium-Priority]** — следующая итерация.
- **[Nit]** — мелкая эстетика.

Описывай **проблему и её влияние**, не предписывай решение. Начинай с того, что работает хорошо. Прикладывай скриншоты к визуальным находкам.
