## 0. TL;DR

Freedom AI Compliance (внутреннее имя — **GCP, Globerce Compliance Platform**) — это внутренняя AML/KYC/комплаенс-платформа для Freedom Bank Kazakhstan. Назначение: автоматический и полуавтоматический мониторинг клиентов и транзакций, генерация оповещений по сценариям и правилам, ведение комплаенс-кейсов, расследование с помощью AI-агентов и ML-моделей, аудит и настройки доступа.

- Локаль: Казахстан, валюта KZT, таймзона Asia/Almaty (UTC+5).
- Язык интерфейса: русский.
- Mock-домен из дизайна: `box-mock.ffb-dev.freedombank.kz` (боевая dev-среда, верифицирована 17.05.2026).
- Ключевые модули: Dashboard, Клиенты, Транзакции, Правила, Оповещения, Кейсы, Конструктор сценариев (`/workflows`), AI-инструменты (`/ai` → Чат, Агенты, Комплаенс-агент), ML-модели (`/instructions`), Использование LLM, Журнал аудита, Настройки. Чат живёт на top-level `/chat`, не привязан к группе сайдбара.
- AI-стек: мульти-модельная оркестрация через Yandex Foundation Models — `deepseek-v32/latest`, `gpt-oss-120b/latest`, `qwen3-235b-a22b-fp8/latest`, `yandexgpt-5.1/latest`, `gemma-3-27b-it/latest`. Внутренние агенты: `agent`, `chat_agent`, `compliance_agent`, `converter`, `edd_full`, `edd_short`, `monitoring_agent`, `search_agent`.
- ML-модели: Bank Offline Feature Store (v1.4), TSAD (v1.2), CTSM — **Compliance Tabular Supervised Model** (v2.0, кодовое имя «Pied Piper»). Оркестрация на Kedro/Airflow, хранилище PostgreSQL + TimescaleDB, скоринг на CatBoost.
- **Compliance Officer AI** — отдельный системный пользователь (`compliance.ai@globerce.local`, role `ai_agent`), автономно обрабатывает алерты и закрывает кейсы. См. §13.

- Стандартные действия: фильтр, сортировка, экспорт, массовые операции, контекстные действия на строке.
- Поддерживаются вкладки внутри карточек (Обзор, История, EDD, Транзакции, Документы, Связанные кейсы, Активность и т. д.).

## 2. Модуль «Мониторинг»

### 2.1 Dashboard
Главный сводный экран:
- Виджеты с KPI: количество активных клиентов, открытых оповещений, кейсов в работе, объём подозрительных транзакций.
- Тренды и графики (день/неделя/месяц).
- Топ-нарушения по сценариям, топ-клиенты по риску.
- Очередь задач: «Мои оповещения», «Мои кейсы».
- Быстрые ссылки на ключевые модули.

### 2.2 Клиенты

#### Список клиентов
Таблица с колонками (как в боевой):
- **Имя** (Cyrillic + Latin транслитерация под именем).
- **ИИН** (для физ. лица) / **БИН** (для юр. лица).
- **Тип**: Физ. лицо / ИП / Юр. лицо.
- **Страна резидентства** (двухбуквенный код: KZ, RU, CN, UA, BY).
- **Риск**: числовой скор (0–100) + категория из `risk_thresholds` (low 0–24, medium 25–49, high 50–74, critical 75+). Бейдж формата `11 — Низкий`, `51 — Средний`.
- **Флаги**: chips короткими кодами (BL, PDL, BLOCKED, VIP).
- **Создано**: дата регистрации.

Фильтры: тип клиента (Все типы / Физ. лицо / ИП / Юр. лицо), сегмент (Все сегменты / Розничный / МСБ / HNWI / Корпоративный), страна, чекбокс «VIP». Поиск по имени, ИИН, email.

#### Карточка клиента

Структурно в боевой админке карточка — **одна длинная scroll-страница** с секциями. В нашем прототипе сохраняем **8 табов** и обогащаем каждый секциями из боевой.

**Шапка:**
- Имя + email + ИИН + Тип + Дата рождения.
- Сайдблок справа: **Канал уведомлений** — 3 чекбокса (Отправка на Email, SMS-сообщение, Push-уведомление на Freedom SuperApp).

**Таб 1 — Обзор:**
- **Профиль · Подробности**: Версия карточки клиента, Колвир-код, Имя, Фамилия, Отчество, Дата регистрации, Срок обслуживания (дн.), Страна резидентства (полное название + код, напр. «Республика Казахстан (KZ / 398)»), Место рождения, Филиал открытия счёта.
- **Флаги** (15 chips с пометками: значение / `Нет` / `—`): VIP, PEP, PDL, EDD, FATCA (гр.), FATCA (нал.), OESR (гр.), OESR (нал.), Представитель, Антифрод, Огранич. счёта, Blacklist, iPDL, nPDL.
- **AI-генерация сценариев**: inline textarea с placeholder «Проверить по санкционным спискам, запустить adverse media, создать алерт…». Cmd+Enter — отправить. Это inline-вход в Compliance Officer AI на контексте клиента.
- **Избранные сценарии**: 4 quick-action пресета — Блокировка клиента / Отчет по документации от клиента / Полная проверка клиента 6 шагов / Санкционная проверка с отправкой на почту. Каждый — play-кнопка. Сайдкарточка «История запусков» (счётчик).
- **Документы клиента** (превью): чипы файлов с тегами `income_proof`, `edd_report`, `employment_letter`, кнопка «Загрузить».

**Таб 2 — Скоринг:**
- **Профиль рисков** — composite score с подскорами от named-агентов:
  - Аномальные транзакции (`0` если транзакций с алертами нет)
  - Внутренний скоринг
  - Новостной агент (числовой `risk_score`, напр. `75`)
  - EDD-агент (числовой `risk_score`)
- Кнопка **Обновить** — пересчёт.
- Итоговый балл с категорией (Низкий / Средний / Высокий / Критический) и таймштамп последнего обновления.

**Таб 3 — Транзакции:** встроенная таблица операций клиента с фильтрами.

**Таб 4 — Оповещения:** список алертов по клиенту.

**Таб 5 — Кейсы:** связанные кейсы (открытые/закрытые).

**Таб 6 — EDD (Enhanced Due Diligence):**
- Существующий чек-лист 6 шагов (Удостоверение / Адрес / Источник средств / Бенефициары / Цель отношений / Доп. документы) с прогресс-баром.
- **Структурированный EDD-отчёт** из 10 секций (как генерирует EDD-агент):
  1. Краткая сводка (Risk Score N/100)
  2. Идентификация субъекта (ФИО, ИИН, дата рождения, юрисдикция, роли, альтернативное написание)
  3. Корпоративная структура (основная компания, БИН, ОКЭД, юр. адрес, бенефициары)
  4. Бизнес-активность (отрасль, продукты, клиенты, география, финансовые показатели)
  5. Регуляторная и судебная история (таблица: Дело · Источник · Дата · Статус · Детали)
  6. Медиа-обзор (таблица: Заголовок · Источник · Дата · URL · Тональность)
  7. Негативные медиа (детальный разбор каждой публикации)
  8. Санкции и списки наблюдения (OFAC, EU, UN — статус)
  9. Цифровой след (соцсети, профили, видео)
  10. Оценка рисков (Положительные находки / Красные флаги / Пробелы в информации / Рекомендация)
- Кнопки `Запустить EDD` и `Скачать PDF`.

**Таб 7 — Новости (Adverse media):**
- **Ключевые выводы** — bullets с резюме.
- Список новостных карточек: Заголовок, Тональность (`POSITIVE` / `NEGATIVE`), Источник, Дата, Summary.
- Кнопка **Обновить** — перезапуск новостного агента.

**Таб 8 — Документы:** полная таблица загруженных документов с тегами.

**Таб 9 — История / Активность:** лог изменений по клиенту (кто, когда, что менял, IP — см. §7.2).

#### EDD
Подробный поток расширенной проверки:
- Чек-листы документов.
- Статусы шагов (запрошено / получено / проверено / отклонено).
- Поле для комментария комплаенс-офицера.
- Возможность поднимать риск / отправлять в кейс.

### 2.3 Транзакции

#### Список транзакций
Колонки (как в боевой):
- **Дата** (без времени в списке; полное время в карточке).
- **Идентификатор** (short UUID hash, напр. `701b2188` или полная форма `7694a2d1-bc09-5db5`).
- **Клиент** (ФИО, кликабельно).
- **Назначение** — код экономической классификации + краткое описание (напр. `119 — Прочие безвозмездные переводы денег физическим лицам`, `111 — Зарплата`, `343 — Переводы между своими счетами в разных банках`, `510 — Финансовый лизинг`, `424 — Покупка/выкуп акций`, `590 — Поступление от иностранного контрагента`).
- **Сумма** в валюте операции (KZT, USD и т.п.).
- **В тенге** — конвертация в KZT по курсу.
- **Комплаенс** — статус: Завершена / Обработана / Авто-отказ / Ожидание / Отклонена.
- **Приоритет** — Низкий / Средний / Высокий.
- **Риск** — Низкий / Средний / Высокий / — (если не оценён).
- **Филиал** — код филиала (`ALM-001` Алматы Центр, `ALA-001`, `ALA-014`, `AST-002` Астана, `SHY-007` Шымкент и др.).

Фильтры: все статусы, все уровни приоритета, все уровни риска, валюта (Currency dropdown).

#### Карточка транзакции
**Шапка:** `Public ID` (полный UUID, copyable) + бейджи `Комплаенс: Завершена` и `Приоритет: MEDIUM`. Ссылка на клиента под заголовком.

**4 верхние карточки в одной строке:**
1. **Информация о транзакции** — Сумма, В тенге, Комиссия, Создана, Изменена, Срок.
2. **Клиент** — ФИО, UUID клиента, кнопка `Открыть досье →`.
3. **Назначение** — Код назначения, Описание, Тип товара (4-digit, напр. `2000`), Описание товара (напр. «Безвозмездный перевод/материальная помощь другому лицу»), Доп. информация (напр. «Возврат долга по расписке от 12.01.2026»).
4. **Участники** — Оператор, ID оператора, Инициатор, Получатель заявки, Маршрут, Требуется одобрение.

**Карточка снизу — Банковская информация:**
- Счёт отправителя, Счёт посредника, Филиал (полное название, напр. «Центральный филиал, Алматы»), Код филиала, Подразделение, Дата ордера, Номер ордера.

**Карточка — Риск-флаги** (chips, напр. «Транзакционная аномалия»).

**Секции снизу:**
- **Оповещения** — счётчик + список с правилом-триггером, описанием и серьёзностью.
- **Связанные кейсы** — список с case-id, статусом и описанием.

### 2.4 Чат (`/chat`, top-level, не в сайдбаре)
Общий AI-чат для офицера комплаенса. **Не пункт сайдбара** в боевой админке — открывается из ⌘K, AI-лендинга (`/ai`) или плавающей кнопки ассистента.

Экран — 2 колонки:
- **Слева** — список диалогов с поиском и кнопкой `+ Новый диалог`. Примеры тайтлов из боевой: «что такое поведенческая проверка», «как рассчитывается профиль рисков?», «почему транзакция 0f0be971 автоматически завершила», «Анализ: income_report_good_7236b0b7.pdf», «Вытащи мне всю информацию доступную по {клиент}». Под каждым тайтлом — дата.
- **Справа** — поле общения с AI-ассистентом + поле ввода `Спросите что-нибудь...` со скрепкой (attachment) и кнопкой Send.

Empty state: «Напишите сообщение чтобы начать диалог».

Ассистент опирается на данные клиентов, транзакций, сценариев, ML-моделей. Используется как «универсальный copilot» для офицера комплаенса.

## 3. Модуль «Расследования»

### 3.1 Правила
Правила = атомарные условия, используемые в workflow-сценариях и алертах.

#### Список правил
Колонки: **Название** (+ описание под названием), **Категория** (Транзакции / Клиенты / Скрининг / Поведение), **Важность** (Критический / Высокий / Средний / Низкий), **Статус** (Активные / Черновик), **Версия** (`v2`, `v3` — правила версионируются), **Обновлено**, **Действия** (Деактивировать / Активировать).

Фильтры: «Все категории» (по типу сущности), «Все статусы» (Активные / Черновик).

#### Создание/редактирование правила
- Название и описание.
- Категория (Транзакции / Клиенты / Скрининг / Поведение).
- Важность (4 уровня).
- **Условия (все должны выполняться)** — AND-группа (OR-группы в боевой нет; стояли в нашем старом плане — отказались, продукт не требует).
- Каждое условие: **Поле** → **Оператор** → **Значение** или **Поле для сравнения**.
- Доступные поля (16): Сумма транзакции, Сумма в тенге, Валюта, Направление, Тип транзакции, Страна контрагента, Риск-скор, KYC статус, PEP, Заблокирован, Тип клиента, Гражданство, Количество транзакций, Средняя сумма транзакций, Задекларированный доход, Ожидаемый базовый уровень.
- Операторы (8): больше `>`, не меньше `>=`, меньше `<`, не больше `<=`, равно `==`, не равно `!=`, входит в список, не входит в список.
- Режим сравнения: **Значение** (литерал) или **Поле** (другое поле) + опциональный множитель `×N` (напр. `Сумма транзакции >= Ожидаемый базовый уровень × 3`).
- Действие при срабатывании: пометить, повысить риск, сгенерировать оповещение, отправить в кейс.

### 3.2 Оповещения (Alerts)

#### Список оповещений
Колонки: чекбокс (для bulk-операций), **Идентификатор** (UUID), **Оповещение** (тайтл `Срабатывание правила` + описание-цитата вида `Rule 'Транзакционная аномалия' triggered by transaction 701b2188-... (1500000.0 KZT)`), **Важность** (Критический / Высокий / Средний / Низкий), **Статус** (Открыто / На проверке / Отклонено / Эскалировано / Закрыто), **Кейс** (бейдж со ссылкой если есть), **Создано**.

Фильтры: Все уровни, Все статусы.

#### Карточка оповещения
- Подробные триггеры: какое правило/сценарий сработало (например, **«Транзакционная аномалия»**), с какими параметрами и значениями.
- Связанная транзакция (с быстрым переходом).
- Профиль клиента (мини-карточка).
- Действия: «принять в работу», «привязать к кейсу», «эскалировать», «закрыть как ложное срабатывание / подтверждённое нарушение».
- Журнал действий по оповещению.

### 3.3 Кейсы

#### Список кейсов
Колонки: **Кейс №** (формат `CASE-YYYYMMDD-XXXXXXXX`, 8-hex sequential суффикс, напр. `CASE-20260506-7F6A3D24`) + краткое описание под ID, **Статус** (Открыто / В работе / На проверке / Эскалировано / Закрыто / Решено), **Клиент** (ФИО или `—` если кейс не привязан), **Исполнитель** (имя или **`Compliance Officer AI`** для авто-кейсов), **Создано**.

Фильтр: Все статусы.

Кнопка справа: `+ Создать кейс`.

#### Карточка кейса
**Шапка:** Case ID (copyable) + статус-badge + ярлык типа (напр. `Авто-кейс: Транзакционная аномалия`) + длинное описание контекста.

**Информация о кейсе — 2 карточки рядом:**
1. **Детали**: Клиент (ссылка), Номер кейса, Создано, Обновлено, Комментариев (count), Доказательств (count), Подзадач (count).
2. **Управление**: Исполнитель (может быть `Compliance Officer AI`), Статус, SLA.

**Связанные оповещения** — collapsible с кнопкой `Перепроверить правила` (для авто-кейсов от Compliance Officer AI).

**Активность** — 4 таба:
- **Комментарии** — список с автором и таймштампом + поле «Написать комментарий…». В авто-кейсах автор многих комментариев — `Compliance Officer AI`.
- **Доказательства** — приложенные файлы и артефакты (OCR-выписки, income proof, скриншоты).
- **Подзадачи** — checklist с дочерними задачами.
- **Хронология** — timeline всех событий по кейсу.

#### Новый кейс (форма)
- Тип кейса (выбор из справочника).
- Клиент (поиск).
- Описание/контекст.
- Приоритет, SLA.
- Привязка существующих оповещений/транзакций.
- Назначение ответственного (может быть `Compliance Officer AI`).

> **Стикер 3:** «Сейчас логика: алерт → кейс → эскалировать → закрыть. А требуется создать флоу кейса для новых тайпов кейсов».
> То есть архитектура должна позволять расширять типы кейсов с разной воркфлоу-логикой, не только классический алерт-кейс. В боевой первый кастомный тип уже есть — `Авто-кейс: Транзакционная аномалия` с workflow `Income Proof Remediation`.

## 4. Модуль «Конструктор сценариев» (`/workflows`)

**Сценарий (workflow)** в боевой админке — не набор условий, а **пайплайн типизированных активностей**, выполняемых последовательно над клиентом. Это исполняемая программа, не предикат.

### 4.1 Вкладки
- **Клиентский** (активный) — пайплайн запускается по одному клиенту.

- **Встроенный** (disabled, «в разработке») — запускается внутри другого процесса или системы.

> **Стикер 4 (обновлено по боевой):** Групповой и Встроенный показаны в UI как заглушки на будущее, активен только Клиентский. Они НЕ удалены — это roadmap-плейсхолдеры.

### 4.2 Список сценариев

Кнопка справа: `+ Создать` → `/workflows/builder`.

### 4.3 Карточка сценария (`/workflows/[id]`)
- Шапка: название + `Тип: Клиентский | ID: #N` + кнопки `Редактировать` / `Удалить`.
- **План** — нумерованный список шагов с типом активности и описанием.
- **Запуск** (правая колонка): поле `user_id` (Select со всеми клиентами) + кнопка `Запустить`.

### 4.4 Редактор сценария — drag-and-drop pipeline (`/workflows/builder?id=N`)

Layout — **3 колонки**:

1. **Доступные активности** (палитра) — 15 типов:
   - Санкционная проверка
   - Поведенческий скоринг
   - OCR документа
   - Заблокировать клиента
   - Отправить email
   - Сформировать отчёт
   - Поиск в новостных источниках

   - Запуск агента (с пользовательской инструкцией)

2. **Пайплайн** — drop-zone с упорядоченным списком шагов: drag handles, нумерация, удаление.

3. **Конфигурация** — панель свойств выбранного шага. Поля зависят от типа активности (для `OCR документа` — выбор doc-tag; для `Отправить email` — to / subject / template; для `Запуск агента` — agent_id и доп. промпт).

DnD-стек в прототипе: `@dnd-kit/core` + `@dnd-kit/sortable`. Mock-state в zustand-store.

> **Стикер 2:** «В избранных сценариях ещё будет запуск WARM (огромный бизнес-процесс проверки клиента) — потребуется просмотр статуса».

## 5. Модуль «AI-Инструменты» (`/ai`)

Лендинг (`/ai`) объединяет три карточки-тайла со ссылками на разделы:
- **Чат** → `/chat`
- **Агенты** → `/agents`
- **Комплаенс-агент** → `/agents/compliance-officer`

Цитата из лендинга: «Все AI-инструменты платформы в одном месте: чат-ассистент, настраиваемые агенты и автоматизированный Комплаенс-агент для обработки оповещений.»

### 5.1 Чат (`/chat`)
См. §2.4. **Top-level раздел, не в сайдбаре**, доступен через ⌘K, AI-лендинг, плавающую кнопку ассистента.

Описание на тайле:

- История диалогов — все разговоры сохраняются и доступны в панели.
- Контекст платформы — ассистент знает о клиентах, транзакциях и делах.

### 5.2 Агенты (`/agents`)
Список пользовательских AI-агентов с кастомными инструкциями. Используются в workflow-сценариях (через активность «Запуск агента»).

3 преднастроенных агента в боевой:
- **Агент по верификации источников дохода** — Верификация источников дохода для KYC/AML.
- **Агент кратких отчётов о соответствии** — Формирует краткий отчёт о соблюдении требований, используя данные скрининга и скоринга.
- **Агент по описанию рисков** — Формирует подробное описание рисков для регуляторной отчётности.

Описание на тайле:
- Кастомные инструкции — полный контроль над поведением агента.
- Интеграция со сценариями — агенты подключаются к сценариям автоматизации.
- Yandex AI — все агенты работают через Yandex Foundation Models.

#### Карточка агента (`/agents/[id]`)
- Имя, описание.
- Системный промпт / инструкции (collapsible block, источник `api/agent_instructions/{name}_instruction.txt`).
- Используемая модель.
- Подключённые инструменты (tools/функции).
- Логи запусков.
- Кнопки edit (pencil) и delete (trash) на карточке списка.

### 5.3 Комплаенс-агент (`/agents/compliance-officer`)
Автономный LLM-агент, реагирующий на новые оповещения и закрывающий кейсы. Это **сердце killer-flow** (§14).

**Шапка:** имя + бейдж `Активен` + кнопка `Выключить`.

**Принцип работы** (6 пронумерованных шагов):
1. Получает каждое новое оповещение сразу после обработки транзакции.
2. Анализирует контекст: правило, транзакцию и клиента — и определяет подходящий сценарий действий.
3. При решении запустить сценарий — создаёт кейс, привязывает оповещение и отправляет клиенту письмо со ссылкой.
4. Ожидает завершения сценария клиентом (до 7 дней) и повторно проверяет правила по обновлённым данным.
5. Если правило больше не срабатывает — кейс закрывается автоматически. Иначе менеджер получает письмо с подробностями.
6. При решении «эскалация» или «пропустить» — отправляет менеджеру письмо или фиксирует решение в журнале.

Примечание под блоком: «Инструкция модели управляется в Yandex AI console; источник в репозитории — `api/agent_instructions/compliance_officer_instruction.txt`».

**Настройки:**
- Email менеджера для отчётов (input).
- Когда отправлять письма — 4 чекбокса:
  - При запуске сценария клиенту.
  - Когда кейс закрыт автоматически.
  - Когда правило всё ещё срабатывает / клиент не ответил.
  - Когда агент эскалирует.

**Активность агента** — таблица:
- Колонки: Время · Действие · Статус · Детали · Связь.
- Типы действий: `Отправлено письмо менеджеру`, `Сценарий завершён`, `Повторная проверка правил`, `Запущен сценарий для клиента`, `Создан кейс`, `Решение LLM` (с детализацией решения).
- Статусы: `ok`, `error`.

Описание на тайле в `/ai`:
- Авто-обработка — агент сам решает что делать с оповещением.
- Эскалация — создаёт кейсы и отправляет письма менеджерам.
- История решений — полный лог действий агента по каждому клиенту.

### 5.4 Плавающая кнопка ассистента
В правом нижнем углу любой страницы внутри `(app)/*` — round button с иконкой ассистента (44px). Открывает Sheet с контекстом текущей страницы (клиент / транзакция / кейс / сценарий). Реюз `AssistantPanel`, монтаж в `app/(app)/layout.tsx`.

### 5.5 Детали агентов (инструкции)
Инструкции агентов оформлены как Markdown-документ:
- Роль и цель.
- Входные данные.
- Правила принятия решений.
- Ограничения (что нельзя делать).
- Формат ответа (структурированные блоки/JSON).

## 7. Модуль «Система»

### 7.1 Использование LLM (`/llm-usage`)

- Всего вызовов (напр. `162`).

- Выходные токены (`219 855`).

## 8. Сквозные доменные понятия

- **Клиент (Customer)** — физ. или юр. лицо с реквизитами, риск-профилем, набором операций, документов и историй EDD.
- **Транзакция (Transaction)** — операция с финансовыми реквизитами, привязанная к клиенту/контрагенту, со сценариями и правилами, которые могли её триггернуть.
- **Правило (Rule)** — атомарное условие над сущностями (клиент/транзакция/группа), используется в сценариях.
- **Сценарий (Scenario)** — комбинация правил + триггеры + действия, может запускать ML-модели и WARM-процессы.
- **Оповещение (Alert)** — событие срабатывания сценария/правила, требующее реакции офицера.
- **Кейс (Case)** — расследование, объединяющее одно или несколько оповещений, транзакций и клиентов.
- **EDD** — расширенная проверка клиента с чек-листами и документами.
- **WARM** — длительный бизнес-процесс комплексной проверки клиента (см. стикер 2).
- **Агент (AI Agent)** — специализированная AI-сущность с промптом, моделью и инструментами.
- **Compliance-агент** — мета-агент, координирующий специализированные агенты.
- **ML-модель** — модель из каталога (Bank Offline FS, TSAD, CTSM и др.), интегрированная в сценарии/скоринг.
- **Риск-фактор** — справочник, влияющий на скоринг и правила.

## 13. Compliance Officer AI — автономный агент

Системный пользователь и LLM-агент, лежащий в основе автоматизации платформы.

### 13.1 Identity

- Email: `compliance.ai@globerce.local`
- Роль: `ai_agent`
- В таблице пользователей статус `Неактивен` (нет UI-сессии — это не интерактивный пользователь, а агент).
- В журнале аудита фигурирует как пользователь `Compliance Officer AI`.
- В кейсах — как `Исполнитель: Compliance Officer AI`.
- В LLM Usage — как агент `compliance_agent`, обычно работающий на модели `deepseek-v32/latest`.

### 13.2 Runtime config

Хранится в `/settings/system` под ключом `compliance_agent_config`:

```json
{
  "enabled": true,
  "manager_email": "tenizbaia@freedombank.kz",
  "send_email_on_dispatch": true,
  "send_email_on_clear": true,
  "send_email_on_fail": true,
  "send_email_on_escalate": true
}
```

Управляется через UI на странице агента (`/agents/compliance-officer`) — там 4 чекбокса «Когда отправлять письма».

### 13.3 Принцип работы (6 шагов)

1. Получает каждое новое оповещение сразу после обработки транзакции.
2. Анализирует контекст: правило, транзакцию и клиента — и определяет подходящий сценарий действий (`dispatch_workflow`).
3. При решении запустить сценарий — создаёт кейс, привязывает оповещение и отправляет клиенту письмо со ссылкой.
4. Ожидает завершения сценария клиентом (до 7 дней) и повторно проверяет правила по обновлённым данным (`Перепроверить правила`).
5. Если правило больше не срабатывает — кейс закрывается автоматически (`COMPLETED`). Иначе менеджер получает письмо с подробностями.
6. При решении «эскалация» или «пропустить» — отправляет менеджеру письмо или фиксирует решение в журнале.

### 13.4 Activity log

На странице агента — таблица «Активность агента»: Время · Действие · Статус (`ok` / `error`) · Детали · Связь (ссылка на кейс/алерт).

Типы действий:
- `Решение LLM` (с детализацией, напр. `dispatch_workflow — Income Proof Remediation...`).
- `Создан кейс` (со ссылкой на CASE-id).
- `Запущен сценарий для клиента`.
- `Отправлено письмо менеджеру` (с заголовком письма).
- `Повторная проверка правил` (с результатом «всё чисто» / «правила всё ещё срабатывают»).
- `Сценарий завершён` (со статусом `COMPLETED` / `FAILED`).

### 13.5 Инструкции и модель

Инструкция управляется в Yandex AI console. Исходник в репозитории — `api/agent_instructions/compliance_officer_instruction.txt`. В UI агента это показывается как note, не редактируемое поле.

Дефолтная модель — `deepseek-v32/latest`. На каждый кейс агент тратит ~3 750 input + ~1 000 output токенов, стоимость ~$0.02, латентность 20–48 секунд.

## 14. Killer flow — центральный нарратив продукта

Это «зачем существует продукт». Прототип должен демонстрировать этот цикл end-to-end на одном предзаряженном клиенте (Дягилев Михаил Владимирович).

### 14.1 Цикл

```
[Транзакция] (1 500 000 KZT, код 119, баланс клиента 0)
    │
    ↓ (правило сработало)
[Alert] Транзакционная аномалия (severity high)
    │
    ↓ (Compliance Officer AI получает алерт)
[Решение LLM] dispatch_workflow → "Income Proof Remediation"
    │
    ↓ (агент создаёт авто-кейс)
[Case CASE-20260506-7F6A3D24] исполнитель = Compliance Officer AI
    │
    ↓ (агент запускает workflow на клиенте)
[Workflow run] Income Proof Remediation
    │
    ↓ (отправлено email клиенту со ссылкой)
[Клиент загружает income_anomaly_remediation.pdf]
    │
    ↓ (OCR извлекает: Средняя выплата 6 240 000 тенге/мес, confidence 0.99)
[Обновление declared_monthly_income]
    │
    ↓ (агент: Повторная проверка правил)
[Rules re-check] всё чисто — правило больше не срабатывает
    │
    ↓ (агент закрывает кейс)
[Case status: Решено] + письмо менеджеру «закрыт автоматически»
```

### 14.2 Артефакты в прототипе

Killer flow «оседает» в следующих местах одновременно:

| Экран | Что видно |
|---|---|
| `/clients/{Дягилев}` | алерт + кейс в табах «Оповещения» и «Кейсы», EDD-агент score 75, Новостной агент score 75 |
| `/transactions/701b2188-...` | транзакция 1.5M KZT, риск-флаг «Транзакционная аномалия», связанный alert и кейс |
| `/alerts/{e0f632de-...}` | алерт severity high, статус «Отклонено» (после авто-разрешения) |
| `/cases/CASE-20260506-7F6A3D24` | статус «Решено», 5 комментариев от Compliance Officer AI, 2 доказательства, 4 таба активности |
| `/workflows/{Income Proof Remediation}` | пайплайн из 6 активностей, история запусков с этим кейсом |
| `/agents/compliance-officer` | в Активности агента — 7 строк по этому кейсу |
| `/audit` (фильтр Кейс) | вся цепочка событий с пользователем `Compliance Officer AI` |
| `/llm-usage` (фильтр `compliance_agent`) | ~10 вызовов deepseek-v32 в этот день |
